# CasesBot

Бот-сборщик кейс-презентаций для отдела продаж (VK Teams + PHP-монолит).
Полное описание задачи, требований и архитектуры — в [`ТЗ_бот-сборщик кейс-презентаций v1.1.md`](./ТЗ_бот-сборщик%20кейс-презентаций%20v1.1.md).

## Структура проекта

```
CasesBot/
├── bin/                # CLI-скрипты: index.php, poll.php (VK Teams), presentations-list.php, build-test.php, deck-test.php
├── config/             # Config, справочник тегов (TagTaxonomy), .env
├── presentations/       # Исходные .pptx — НЕ в git (см. .gitignore), заливаются на сервер отдельно при деплое
├── public/              # Точка входа для вебхука VK Teams
├── python/              # SlideTextExtractor / SlideCloner (python-pptx, lxml) — вызываются из PHP как подпроцесс
├── src/
│   ├── Api/Providers/   # LLM-провайдеры (Ollama, OpenCodeZen) для дополнения тегов кейсов
│   ├── Bot/             # ChatBotController (диспетчер), Commands/ (по классу на команду), VkTeamsClient
│   ├── Catalog/         # CaseItem, CatalogRepository, TagTaxonomy, Indexer, LlmTagSuggester
│   ├── Query/           # QueryParser, Matcher
│   ├── Presentation/    # SlideTextExtractor, SlideCloner, PresentationBuilder (PHP-обёртки)
│   ├── Storage/         # LocalPresentationsClient (временно вместо Google Drive)
│   └── Config.php
├── storage/
│   ├── catalog/         # Каталог кейсов: catalog.sqlite + schema.sql + images/ (картинки со слайдов)
│   ├── incoming/        # Резерв на будущее (если источник вернётся на скачивание, напр. Google Drive)
│   ├── output/          # Собранные pptx перед отправкой в чат
│   └── logs/            # Журнал запросов (P1)
├── tests/
│   ├── Unit/
│   └── Fixtures/        # Тестовые pptx для проверки сохранения форматирования
└── docs/                # Дополнительная документация по мере появления
```

## Индексация каталога

`php bin/index.php` обходит `presentations/`, извлекает текст и картинки слайдов
(`python/slide_text_extractor.py`), но только для слайдов, помеченных экспертом меткой `#кейс#`
в заметках докладчика (не видна ни на слайде, ни в показе) — остальное (обложка, о компании,
контакты) пропускается автоматически.

Предложенные теги для помеченных слайдов собираются из трёх источников: (1) теги, которые эксперт
сам перечислил в тех же заметках после метки (`#кейс#\n1с, ритейл`) — им доверяем больше всего;
(2) LLM по содержимому слайда (`LlmTagSuggester` через `src/Api/Providers`, провайдер и модель —
в `.env`: `AI_PROVIDER`, `OLLAMA_*`/`OPENCODEZEN_*`); (3) простое совпадение по словарю
`config/tags.php`. Теги плоские, без категорий (§6.2 ТЗ изначально предполагал
industry/product/technology/client — от категорий отказались в пользу простых тегов). Эксперт
подтверждает/правит итог за один проход (Enter — принять предложенное, `skip` — пропустить, свои
теги — через запятую), плюс необязательную ссылку на сайт (`site_url`) — LLM дополняет подсказку,
не решает за эксперта (§5.1.8 ТЗ).

Полный текст сохраняется в `cases.content`, картинки — файлами в `storage/catalog/images/` со
ссылками в `case_images`. Повторный запуск обновляет те же записи, не создавая дублей.

## Сборка презентации

`SlideCloner` (`python/slide_cloner.py`) копирует слайды между презентациями на уровне сырых частей
OOXML-пакета (python-pptx opc/package API, не через объектную модель) — слайд, его layout, master,
theme и все media копируются как есть, с ремаппингом id/media/layout-master; заметки докладчика
(там могут быть внутренние `#кейс#`/теги эксперта) не копируются. Проверено: форматирование
(кастомные шрифты, цвета, размеры) сохраняется побайтово идентично оригиналу.

`PresentationBuilder` вызывает `SlideCloner` для нужных слайдов, затем сам добавляет титульный слайд
(тема + дата) и сохраняет финальный файл в `storage/output/`.

`CaseDeckBuilder` — другой сценарий сборки, для команды `кейсы`: из исходной презентации убирает
все слайды с меткой `#кейс#` в заметках докладчика, кроме тех номеров, что нашлись по запросу
(`SlideTextExtractor` определяет метку, `SlideCloner` копирует нужные слайды в исходном порядке).
Титульный слайд не добавляется — получившийся файл читается как урезанная версия оригинала.
PDF как формат вывода не используется — LibreOffice headless на Alpine/musl (текущий базовый образ)
падает на старте (баг совместимости, проверено на сервере), поэтому отправляется pptx.

Проверка `CaseDeckBuilder` без чат-интеграции:
```
php bin/deck-test.php "Экспертиза направления бизнес автоматизации.pptx" "7,9"
```

## Команды бота

Команды — отдельные классы в `src/Bot/Commands/`, реализующие `CommandInterface` (`matches()` +
`handle()`); `ChatBotController` перебирает их и передаёт управление первой подошедшей. Добавить
новую команду — значит просто добавить новый файл в эту папку и зарегистрировать его в `bin/poll.php`.

**`кейсы`** (`CasesCommand`) — ищет и ранжирует кейсы по тегам (`CatalogRepository::findByTags`,
больше совпадений — выше, лимит `MAX_SLIDES_PER_DECK`, §5.1.3 ТЗ). Теги — простые слова через
запятую/пробел (без категорий), с резервом на синонимы (`TagTaxonomy`, `config/tags.php`) —
свободный текст всей фразы целиком (`QueryParser`) ещё не реализован:
```
кейсы ритейл
кейсы производство, 1с
```
По каждой затронутой исходной презентации собирает и отправляет pptx-подборку (`CaseDeckBuilder`,
см. «Сборка презентации» выше), а следом — в любом случае — ссылку на оригинал презентации
целиком и номера слайдов, которые нужно оставить, если подборку нужно поправить вручную. Если
сборка не удалась (или `CaseDeckBuilder` не настроен) — присылается только ссылка.

Если ничего не найдено — бот присылает список реально используемых в каталоге тегов вместо пустого
файла (§5.1.7 ТЗ).

Если же ни одно слово запроса вообще не совпало ни с одним известным тегом (ни в `TagTaxonomy`, ни
в базе), а LLM-провайдер настроен (`.env`: `AI_PROVIDER`, `OLLAMA_*`/`OPENCODEZEN_*`), команда не
сдаётся сразу с подсказкой по тегам, а отдаёт запрос целиком и текст всех видимых кейсов каталога
(`CatalogRepository::allForMatching` — текст уже лежит в `cases.content` с момента индексации,
повторно читать презентации не нужно) в `LlmCaseMatcher`. Модель либо возвращает id кейсов,
подходящих по смыслу (отрасль/продукт/технология/клиент), либо явно отвечает, что подходящих нет —
тогда бот показывает тот же список доступных тегов, что и при обычном пустом результате. Без
настроенного провайдера поведение прежнее — подсказка с примерами синтаксиса.

Проверка без чат-интеграции — то же самое, что делает команда, но из консоли:
```
php bin/build-test.php "производство, 1с"
```

## Этапы разработки (§9 ТЗ)

1. **Недели 1–2** — `Indexer`, `CatalogRepository`, `TagTaxonomy`: построение каталога, проверка тегирования/поиска через `bin/index.php`. ✅
2. **Недели 3–4** — `SlideCloner`, `PresentationBuilder`: сборка pptx (готово, проверено на форматировании) ✅; команда `кейсы` в VK Teams ✅ — но по явным тегам, не свободному тексту (`QueryParser` с синонимами ещё нет).
3. **Неделя 5+** — пилот на реальных запросах, донастройка справочника тегов и лимитов.
